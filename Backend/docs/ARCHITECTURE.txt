SMARTSHIP LOGISTICS — ARCHITECTURE

1) Architecture style
- Microservices architecture with an API Gateway (Ocelot) as the single entry point for the frontend.
- Shared contracts library for integration events.
- Mix of synchronous HTTP calls (via gateway) and asynchronous event-driven updates (RabbitMQ).

2) Logical view

[Angular UI]
   |
   | HTTP (via dev proxy to gateway)
   v
[Ocelot Gateway]  (Swagger aggregation + routing)
   |
   | routes to services
   v
[Identity] [Shipment] [Tracking] [Document] [Admin]
   |
   | EF Core
   v
[SQL Server]

Asynchronous path:
[Shipment/Admin] -- publish IntegrationEvent --> [RabbitMQ exchange: smartship_events]
                                               |
                                               v
                                      [TrackingService consumer]

3) Gateway routing (current policy)
The gateway (gateway/SmartShip.Gateway/ocelot.json) exposes a clean upstream surface:
- /gateway/auth/{everything} -> IdentityService /api/auth/{everything}
- /gateway/identity/Profile and /gateway/identity/Profile/{everything} -> IdentityService Profile APIs
- /gateway/identity/Roles and /gateway/identity/Roles/{everything} -> IdentityService Roles APIs
- /gateway/identity/Users and /gateway/identity/Users/{everything} -> IdentityService Users APIs
- /gateway/identity/admin/{everything} -> IdentityService admin APIs
- /gateway/shipments/{everything} -> ShipmentService /api/shipments/{everything}
- /gateway/tracking/{everything} -> TrackingService /api/{everything}
- /gateway/documents/{everything} -> DocumentService /api/documents/{everything}
- /gateway/admin/{everything} -> AdminService /api/{everything}
- /uploads/{everything} -> DocumentService /uploads/{everything}

Key intent:
- No broad /gateway/identity/{everything} passthrough
- Auth routes are exposed only under /gateway/auth/*

4) Service responsibilities

IdentityService
- Auth endpoints under /api/auth/* (login, signup/otp, refresh, password flows)
- Identity domain endpoints under /api/Profile, /api/Users, /api/Roles, plus /api/admin/*
- JWT issuance and role-based authorization support

ShipmentService
- Shipment lifecycle APIs under /api/shipments/*
- Publishes integration events (IntegrationEvent subclasses) to RabbitMQ

TrackingService
- Tracking APIs (including anonymous-friendly tracking retrieval)
- BackgroundService consumer: subscribes to RabbitMQ events and updates tracking status/history

DocumentService
- Document upload endpoints under /api/documents/*
- Static hosting for uploaded files under /uploads/* (gateway forwards /uploads)

AdminService
- Admin-only APIs under /api/*
- Includes dashboard stats, hubs/locations CRUD, reports, and exception workflows

5) Cross-cutting concerns
- AuthN/AuthZ: JWT bearer auth, role checks on controllers/actions
- Logging: Serilog configured per service
- Persistence: EF Core migrations on startup pattern (service Program.cs)
- Contracts: SmartShip.Contracts.Events used by publisher and consumers

6) Deployment model (local)
- docker-compose can run SQL Server + RabbitMQ + the APIs (gateway + services)
- Angular dev server proxies to the gateway for a single-origin dev experience
