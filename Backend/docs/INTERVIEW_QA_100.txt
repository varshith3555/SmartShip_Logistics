SMARTSHIP LOGISTICS — INTERVIEW Q&A (100)

1) Q: What is this system’s main purpose?
   A: Manage shipments, tracking, documents, and admin operations behind a single gateway for a logistics platform.

2) Q: What is the main client?
   A: An Angular 17 web application.

3) Q: Why Angular Material?
   A: To speed up consistent UI delivery with a robust component library.

4) Q: How does the UI reach the backend locally?
    A: Angular dev proxy forwards /gateway and /uploads to the gateway at http://localhost:5000.

5) Q: What is the single external API surface?
   A: The Ocelot gateway.

6) Q: What do you gain by keeping a stable /gateway/* convention?
   A: Client code stays simple; service changes are isolated behind gateway routing.

7) Q: Which services exist?
   A: IdentityService, ShipmentService, TrackingService, DocumentService, AdminService.

8) Q: What is the gateway’s target framework?
   A: net10.0.

9) Q: What do the services target?
   A: net8.0.

10) Q: How are endpoints documented?
    A: Each service provides Swagger; the gateway aggregates via SwaggerForOcelot.

11) Q: How do you aggregate multiple swagger documents?
    A: In ocelot.json, each route uses a SwaggerKey and SwaggerEndPoints maps keys to swagger.json URLs.

12) Q: What’s the key gateway design choice for identity routes?
    A: Expose auth via /gateway/auth/* and restrict identity management to explicit Profile/Users/Roles/admin routes.

13) Q: Why avoid a catch-all identity route?
    A: To prevent accidental exposure and control the public surface precisely.

14) Q: What pattern is used for persistence?
    A: EF Core DbContext per service with SQL Server.

15) Q: Why per-service databases/models?
    A: Reduces coupling and allows services to evolve independently.

16) Q: How do you handle migrations?
    A: Services commonly apply EF Core migrations on startup (Program.cs pattern).

17) Q: How do you handle auth?
    A: JWT bearer authentication with role-based [Authorize].

18) Q: Where are tokens stored on the client?
    A: sessionStorage (access token + refresh token).

19) Q: What is the refresh strategy?
    A: UI calls /gateway/auth/refresh; if refresh fails it clears tokens and logs out.

20) Q: What is the messaging broker?
    A: RabbitMQ.

21) Q: What exchange type is used?
    A: Direct exchange.

22) Q: What is the exchange name?
    A: smartship_events.

23) Q: What is the event publishing convention?
    A: routingKey = .NET event type name.

24) Q: Where is the publisher implementation?
    A: SmartShip.Core.Messaging.RabbitMQService implements IEventBus.

25) Q: What does IEventBus contain?
    A: A Publish(IntegrationEvent) method.

26) Q: Why use a messaging abstraction?
    A: Easier to swap implementations and keeps services from depending on RabbitMQ details everywhere.

27) Q: Which service consumes events?
    A: TrackingService.

28) Q: How is the consumer hosted?
    A: As a BackgroundService (TrackingEventConsumer).

29) Q: What queue is used?
    A: tracking_queue.

30) Q: Which routing keys are bound to the queue?
    A: ShipmentCreatedEvent, ShipmentBookedEvent, ShipmentDeliveredEvent, TrackingUpdatedEvent.

31) Q: What does ShipmentCreatedEvent processing do?
    A: Creates/updates shipment status to CREATED and adds a history entry.

32) Q: What does TrackingUpdatedEvent processing do?
    A: Updates current status and adds a history record with location/remarks.

33) Q: Why process based on routingKey?
    A: To choose the correct event DTO to deserialize and the correct handler logic.

34) Q: How do you ensure messages aren’t lost?
    A: Use durable queue declarations and ACK after processing.

35) Q: What’s an idempotent consumer and why care?
    A: It tolerates duplicate deliveries without corrupting state; important for reliability.

36) Q: What HTTP routes does the UI call for shipments?
    A: /gateway/shipments/{everything}.

37) Q: What do shipments routes map to downstream?
    A: ShipmentService /api/shipments/{everything}.

38) Q: How does tracking routing look?
    A: /gateway/tracking/{everything} -> TrackingService /api/{everything}.

39) Q: How do documents routes look?
    A: /gateway/documents/{everything} -> DocumentService /api/documents/{everything}.

40) Q: How are uploaded files served?
    A: Through /uploads/{everything} forwarded to DocumentService /uploads/{everything}.

41) Q: Why separate /uploads from /gateway/documents?
    A: Separates API calls from static content delivery paths.

42) Q: Name DocumentService upload endpoints.
    A: upload, upload-invoice, upload-label, upload-customs.

43) Q: How do you fetch docs for a shipment?
    A: GET /gateway/documents/shipment/{shipmentId}.

44) Q: What is delivery proof?
    A: Admin uploads proof (image + signature) via /gateway/documents/delivery-proof/{shipmentId}.

45) Q: What does AdminService expose?
    A: /api endpoints for dashboard, exceptions workflows, hubs/locations CRUD, reports.

46) Q: What is an example admin dashboard endpoint?
    A: GET /gateway/admin/dashboard.

47) Q: What are exceptions in this context?
    A: Shipment issues requiring admin actions like resolve, delay, return.

48) Q: Example exception action route?
    A: PUT /gateway/admin/shipments/{id}/resolve.

49) Q: Example reports routes?
    A: /gateway/admin/reports/shipment-performance, /delivery-sla, /revenue, /hub-performance.

50) Q: What logging library is used?
    A: Serilog.

51) Q: What’s the benefit of Serilog?
    A: Structured logs and flexible sinks (console/file) for easier troubleshooting.

52) Q: How do you debug routing problems?
    A: Validate ocelot.json templates, confirm downstream base paths, check gateway logs, and test service swagger endpoints.

53) Q: How do you debug authentication issues?
    A: Confirm JWT claims/roles, check token expiry, verify Authorization headers, and confirm gateway routes map to correct downstream.

54) Q: Where is auth implemented on the client?
    A: AuthService uses ApiClient to call /gateway/auth/* and stores tokens.

55) Q: How is role determined client-side?
    A: By decoding the JWT payload and reading the role claim.

56) Q: What’s a potential security improvement?
    A: Store tokens more securely (consider httpOnly cookies) and apply tighter CORS and rate limiting.

57) Q: Why keep contracts in SmartShip.Contracts?
    A: Producers and consumers share strongly-typed event definitions and avoid schema mismatch.

58) Q: How does the gateway handle method forwarding?
    A: Routes specify allowed upstream HTTP methods (GET/POST/PUT/DELETE/etc.).

59) Q: Why include OPTIONS in gateway routes?
    A: To support CORS preflight requests.

60) Q: What’s the difference between /gateway/auth/* and /gateway/identity/*?
    A: /gateway/auth/* covers authentication flows; /gateway/identity/* covers profile/users/roles/admin identity management.

61) Q: How does document upload authentication work?
    A: DocumentsController requires [Authorize] for uploads; some GET endpoints allow anonymous.

62) Q: What endpoints are anonymous in documents?
    A: Get by id, get by shipment, get delivery proof (per controller attributes).

63) Q: What endpoints are admin-only in documents?
    A: Update/delete documents and delivery-proof upload.

64) Q: What’s a good reliability practice for event publishing?
    A: Add retry/backoff and confirm-publish where appropriate; log failures.

65) Q: What’s a good reliability practice for event consumption?
    A: Dead-letter queues, retries, and poison message handling.

66) Q: What’s the biggest performance hotspot risk?
    A: Gateway bottleneck or heavy DB queries; mitigate with caching, paging, and scaling.

67) Q: How do you scale services?
    A: Horizontally scale stateless APIs; scale DB and RabbitMQ appropriately.

68) Q: How do you keep services from tightly coupling to each other?
    A: Use events for cross-service propagation and keep synchronous calls limited to the gateway surface.

69) Q: What is eventual consistency here?
    A: Tracking updates may arrive slightly after shipment creation; the tracking read model catches up via events.

70) Q: Why is tracking a separate read model?
    A: Optimized for query patterns (status + timeline) and built from integration events.

71) Q: How do you handle “same tracking number” updates?
    A: Upsert current status and append history events for timeline.

72) Q: What’s the biggest code maintainability improvement you’d add?
    A: Centralized error handling, consistent DTOs, and stronger API versioning strategy.

73) Q: How do you ensure route naming consistency?
    A: Keep a single convention in ocelot.json and align service controller routes to /api/* patterns.

74) Q: What’s your approach to API documentation?
    A: Swagger per service + gateway aggregation for a single dev entry point.

75) Q: What is the “uploads” path used for?
    A: Serving static files generated by DocumentService.

76) Q: How do you keep file URLs stable?
    A: Use /uploads/* upstream as the stable entry, regardless of DocumentService host/port.

77) Q: What’s the advantage of having an AdminService separate?
    A: Admin domain can evolve independently; access control and scaling are isolated.

78) Q: What’s a common Windows dev issue you saw?
    A: Rebuild failures when the gateway exe is locked by a running process.

79) Q: How would you verify RabbitMQ wiring?
    A: Check exchange/queue/bindings in RabbitMQ UI and confirm routingKey matches event type name.

80) Q: How would you verify Swagger aggregation?
    A: Confirm SwaggerEndPoints URLs are reachable and gateway swagger UI loads all groups.

81) Q: What improvements would you make to tracking consumer coverage?
    A: Implement handlers for all bound event types (booked/delivered) in addition to created/updated.

82) Q: What’s a careful approach to schema changes in events?
    A: Add new optional fields; avoid removing/renaming fields without versioning.

83) Q: Where is the core messaging code?
    A: shared/SmartShip.Core/Messaging.

84) Q: How do you test message-driven code?
    A: Integration tests with a RabbitMQ container or contract tests for event payloads.

85) Q: How do you test gateway routes?
    A: Smoke tests calling /gateway/* and verifying downstream service responses.

86) Q: What is the gateway base URL in config?
    A: GlobalConfiguration.BaseUrl is http://localhost:5000.

87) Q: What’s the advantage of “SwaggerKey” per route?
    A: It groups upstream endpoints to the correct downstream swagger doc in the aggregated UI.

88) Q: What HTTP scheme is used for downstream?
    A: http in local configuration.

89) Q: How do you handle preflight requests?
    A: Include OPTIONS in allowed upstream methods and configure CORS in gateway.

90) Q: Why put /api prefix in service routes?
    A: Consistent API base path per service; easy gateway mapping.

91) Q: How is DocumentsController routed?
    A: [Route("api/[controller]")] -> /api/documents/*.

92) Q: What authentication attribute is used there?
    A: [Authorize] on controller, with some actions [AllowAnonymous] or [Authorize(Roles=...)] overrides.

93) Q: What is the AdminController base route?
    A: [Route("api")] -> /api/*.

94) Q: How do you protect admin endpoints?
    A: [Authorize(Roles = "ADMIN")] at controller level.

95) Q: Why keep the docs in /docs?
    A: Easy to share, review, and update without touching code.

96) Q: What’s your biggest architectural “sell”?
    A: A curated gateway API surface + event-driven tracking read model.

97) Q: What is a good explanation of the tracking design?
    A: “Tracking is updated asynchronously from shipment events and exposes query-friendly endpoints for status and timeline.”

98) Q: What’s one thing you’d add for production readiness?
    A: Health checks, centralized tracing, and stronger resilience policies (retries/circuit breakers).

99) Q: What’s one thing you’d add for security?
    A: Rate limiting at gateway and improved token storage strategy.

100) Q: What’s the cleanest end-to-end story?
     A: “Login -> create shipment -> event publish -> tracking consumer updates timeline -> user sees tracking progress; plus documents upload served via /uploads.”
